# ZeroError: End-to-End Technical Architecture

The ZeroError Cognitive Adaptive Framework is built horizontally across five interconnected microservices orchestrating everything from Next.js Server Components to Python Causal Inference Pipelines. 

This document outlines the full end-to-end technical life cycle.

## 1. Environment & Stack Architecture
The system is constructed within a Docker Compose ecosystem housing the following 5 distinct containers:
1. **PostgreSQL 16:** The unified operational database.
2. **Storefront (Next.js 14):** Port 3000. Customer facing UI.
3. **Analytics API (Node.js/Express):** Port 3001. Event ingestion and scoring backend.
4. **Dashboard (Next.js 14):** Port 3002. Internal analytical visualization.
5. **Analytics Engine (Python/FastAPI):** Background causal processing and calculation.

## 2. Storefront (Next.js App Router)
The frontend establishes the user context and integrates with the tracking ecosystem.
- **State Configuration:** Managed globally via context providers wrapped in `app/layout.tsx` (`AuthProvider`, `WishlistProvider`, `CartProvider`).
- **Simulated Database & Auth Constraints:** This environment explicitly utilizes a Mock Database stored in LocalStorage (`MOCK_DB_KEY`). We chose to forego a hard PostgreSQL password database at this stage to prevent the unnecessary exposure of sensitive user data when deployed openly on cloud platforms (e.g., Vercel) for large-scale behavioral data-gathering studies. Instead of backend password hashing, we rely on rigid front-end sanitization logic.
- **Frontend Form Hardening:** Forms located within the Checkout sequences (`app/checkout/page.tsx`) and Identity creation (`app/signin/page.tsx`) explicitly restrict inputs. Keystrokes are physically bounded by RegExp masks, ensuring numbers cannot penetrate alphabetical input limits, reducing Cross-Site Scripting (XSS) attack vectors on the client side.
- **Intervention Provider:** Hooks into the frontend render lifecycle, querying the Analytics API for real-time risk scores and applying real-time dynamic UI components accordingly.

## 3. The Tracker (`lib/tracking/tracker.ts`)
The `Tracker` class handles the asynchronous, high-frequency capture of telemetry data.
- **Micro-Events:** Captures scroll velocity, hover duration, and explicit micro-hesitations.
- **Batching:** Implements an internal queue and timer (`flushTimer`) to minimize API network requests by POSTing telemetry payloads in bulk chunks to the Analytics API.
- **Binding:** Directly injects the `session_id` and `user_id` into the payload of every DOM interaction.

## 4. Analytics Database Schema (PostgreSQL)
A centralized SQL schema enables data continuity completely un-siloed between components:
- `events`: Captures all raw tracker payloads.
- `contextual_features`: Aggregates events into calculated behavioral vectors (e.g., page views count, hesitation sequences).
- `interventions`: Logs when a user was assigned to an experimental treatment group.
- `rewards`: Links backfilled downstream events (e.g., `purchase` success) back to previously triggered interventions.

## 5. Analytics API (Ingestion & Scoring)
This Node.js/Express abstraction acts as the system's core traffic controller.
- **Event Intake:** The `/api/events` route ingests tracker bulk payloads, safely transforming and UPSERTing telemetry into the database.
- **Reward Backfilling:** An internal listener that intercepts `purchase` events. It walks backward over the associated `session_id` to locate previous `intervention` records and patches them with a `.reward = 1.0`, effectively closing the causal loop.
- **Dynamic Risk Evaluation:** The `/api/score` endpoint determines whether a user is targeted for an intervention. To ensure algorithmic integrity, this uses a scientifically sound *fixed-probability randomized controlled experiment* logic (e.g., uniform random assignment across control/treatment groups).

## 6. Causal Analytics Engine (Python Model)
The statistical heart of the system operates asynchronously.
- **Implementation:** Python processing engine. Runs initially on start-up and re-evaluates strictly on interval sequences via cron/bash polling loops.
- **Engine Logic:** Queries the `interventions`, `rewards`, and `contextual_features` tables. 
- **Methodology (IPW/CATE):** Utilizes causal inference models—specifically Inverse Probability Weighting (IPW) and Conditional Average Treatment Effects (CATE). This controls for observational bias and outputs an explicit evaluation determining the true causal "Uplift" (in units of conversion rate percentage) generated by each interface change. 

## 7. Command Center Dashboard (Next.js)
The visualization layer mapping Engine output natively to UI telemetry.
- **System Monitoring:** Connects natively to the Analytics API to query aggregated event traffic.
- **Effectiveness Visualization:** Displays the resulting CATE estimates mapped out by the Python engine, allowing administrators to definitively decide if a Storefront modification is mathematically proven to generate user conversion.

## Infrastructure & Scaling

### Database Architecture
The backend analytics engine (`analytics-api`) relies on **PostgreSQL**. Since Vercel uses ephemeral serverless functions, local file-based databases like SQLite will fail on deployment. 
To deploy this project to Vercel free-of-charge:
1. Provision a free **Vercel Postgres** or **Supabase** instance via the Vercel Dashboard.
2. Link the project to automatically populate `POSTGRES_URL` in your Environment Variables.
3. Use the `npm run db:init` command to automatically trigger `scripts/init-db.js`, which formats the cloud database with the correct relational schemas (`users`, `sessions`, `events`, `ab_assignments`) using native `pg` connectors.

### CI/CD and Bootstrapping Procedure (Local Docker Environment)
If developing locally instead of using Vercel remotely, the entire stack is configured to boot reliably via Docker Compose without external dependencies.
1. `docker-compose up --build -d` provisions all 5 systems.
2. The `postgres` container executes schema and seed SQL injections locally.
3. The Storefront assumes `http://localhost:3001` telemetry targets natively configured through environmental configurations.
